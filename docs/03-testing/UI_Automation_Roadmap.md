# UI 自动化测试规划建议

## 结论先说

建议做，但不建议一步到位做“全量像素级截图回归”。

对当前这个客户端，更合适的路线是：

1. 先保留现有 GUT API 集成测试作为业务主干
2. 再补一层“UI 结构断言 + 多分辨率 smoke”
3. 等样式稳定后，再考虑截图对比或真机自动化

也就是说，后续值得建立“标准的 UI 自动化测试流程”，但推荐分阶段推进。

## 为什么现在值得做

近期客户端已经出现以下类型的 UI 变更：

- 全局字体 / 图标替换
- TabBar 与子 Tab 样式模板化
- 展示面板模板化
- 弹窗模板化
- 设置页重构
- 炼丹坊 / 储纳 / 术法等页面持续做视觉微调
- 长屏 / 安全区 / 异形屏适配

这类改动的共同特点是：

- 业务逻辑未必出错
- 但控件位置、可见性、点击热区、分辨率下观感很容易悄悄跑掉

单靠 API/GUT 业务测试，不足以覆盖这类风险。

## 不建议直接上的方案

### 1. 全量截图像素对比

现在不建议马上上：

- 字体渲染在不同平台差异大
- Godot 编辑器运行、独立窗口、不同系统缩放都会引入噪声
- 当前 UI 仍在高频微调，基线图很快失效

### 2. 直接上真机端到端自动点击

现在也不建议作为第一阶段主线：

- 接入成本高
- 维护成本高
- 每次 UI 调整后，坐标和流程都容易变

## 建议的标准流程

### 第一阶段：结构型 UI smoke

目标：低成本发现明显跑版和节点缺失。

建议新增一类 GUT/UI smoke，用例只检查：

- 关键节点存在
- 关键节点可见
- 关键节点位于安全可视区域内
- 关键容器尺寸满足最小阈值
- 弹窗居中且不越界

适合覆盖：

- 顶部栏
- 底部主 Tab
- 内视子 Tab
- 属性面板
- 突破面板
- 储纳详情卡
- 设置页关键模块
- 术法弹窗
- 账号编辑弹窗

### 第二阶段：多分辨率 UI smoke

目标：验证主要页面在核心预设分辨率下不越界、不遮挡。

建议先固定三组尺寸：

- `1080×1920`
- `1080×2400`
- `1125×2436`

可以基于当前 [ResolutionPreview.tscn](/Users/hsams/Documents/idle_cultivation_project/idle_cultivation_client/scenes/debug/ResolutionPreview.tscn) 的思路，新增 headless / script 驱动 smoke：

- 切视口尺寸
- 加载登录或主场景
- 断言关键节点矩形都落在可视区 / 安全区内

### 第三阶段：关键截图基线

等 UI 风格阶段性稳定后，再只给少数核心页面上截图对比：

- 登录页
- 主界面内视页
- 储纳页
- 设置页
- 术法详情弹窗

截图回归不需要全项目覆盖，只做高价值页面即可。

## 当前建议优先补哪些自动化项

### 高优先级

- 主界面内视页
  - `NeishiTabBar`、属性面板、小人、突破面板、日志框都在安全可视区内
- 顶部栏 / 底部栏
  - 不越界
  - 安全区下仍保留最小点击区域
- 术法详情弹窗
  - 弹窗居中
  - 不压安全区
  - 外层遮罩存在
- 账号编辑弹窗
  - 昵称输入框、头像网格、提交按钮均可见

### 中优先级

- 储纳页
  - 背包格子区域、详情卡、按钮都在可视区
- 设置页
  - FPS 区、音频区、兑换码区、按钮列表不越界
- 炼丹坊
  - 丹方列表、工作区、炼丹加成三块布局完整

### 低优先级

- 细粒度的像素一致性
- 字体抗锯齿或单字符渲染回归

## 推荐落地形式

### 方案 A：继续基于 GUT 扩展

优点：

- 与现有测试体系一致
- 最容易接入
- 能继续复用现有测试启动脚本

建议新增目录：

- `tests_gut/ui_layout/`

建议新增能力：

- 统一的节点矩形断言 helper
- 统一的安全区断言 helper
- 多分辨率测试启动 helper

### 方案 B：额外做一个轻量 UI probe 脚本层

优点：

- 可以更接近运行时场景
- 可复用给人工调试和自动 smoke

建议做法：

- 保留 `ResolutionPreview` 作为人工工具
- 再补一个纯脚本 smoke scene，用于 headless 检查节点矩形

## 是否现在就做

建议：

- 现在可以开始做第一阶段和第二阶段
- 第三阶段截图回归先不急

原因：

- 第一、二阶段投入较小，能明显降低 UI 跑版风险
- 当前项目 UI 模板和安全区结构刚建立，正是补“结构型自动化”的合适时机
- 截图回归要等这轮视觉风格更稳定后再上，否则维护成本太高

## 推荐的下一批实施顺序

1. 抽一个公共 UI 布局断言 helper
2. 给主界面、术法弹窗、账号编辑弹窗补 3~5 条结构型 smoke
3. 给 `1080×1920 / 1080×2400 / 1125×2436` 三组分辨率补多分辨率 smoke
4. 观察一轮维护成本，再决定要不要推进截图基线

## 当前结论

后续应该建立标准 UI 自动化测试流程，但推荐路线是：

- 先做结构型、多分辨率的 UI smoke
- 暂缓全量截图对比
- 暂缓真机端到端自动点击
