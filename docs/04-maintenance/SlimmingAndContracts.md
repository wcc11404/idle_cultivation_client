# 瘦身与契约维护规范

## 瘦身原则

- 不改变现有体验链路：修炼乐观更新、炼丹预扣、历练回放。
- 删除“确定无调用”的旧逻辑优先；存疑项先隔离再删。
- 每批清理后执行 `./run_tests.sh` 回归。

## 外部注入契约字段

部分字段可能在模块内不直接读取，但由 `GameUI` 或初始化流程注入，视为契约字段。

- 处理原则：不能仅因“本文件内无引用”而删除。
- 只有在同步改造调用方并完成回归后，才能下线。

## 文案与错误处理规范

- 业务提示统一由模块根据 `reason_code + reason_data` 翻译。
- 技术错误（网络/鉴权/服务器异常）由网络层统一处理。
- 禁止恢复到“直接透传服务端业务 message”模式。

## 数值显示格式契约

- 客户端统一优先使用 `res://scripts/utils/ui_utils.gd` 内的格式化函数，不在模块内直接手写 `str(...) / %d / %.2f` 来拼主要业务数值。
- 当前全局有两套正式口径：
  - `UIUtils.format_display_number(...)`
    - `<1000`：保留两位小数并去尾零
    - `>=1000`：转 `K / M / B`，保留一位小数并去尾零
  - `UIUtils.format_display_number_integer(...)`
    - `<1000`：保留整数
    - `>=1000`：转 `K / M / B`，保留整数
- 当前约定：
  - 属性面板与战斗页的气血/灵气：走整数口径
  - 主界面顶部灵石、排行榜灵气、攻击/防御/速度/灵气获取速度、战斗伤害：走非整数口径
  - 容量、槽位、剩余次数、等级提升结果、格子数量等“计数型文案”：优先走整数口径
  - 百分比、时间、中文层级（如 `第x层` / `Lv.`）属于特殊展示，不强制套统一压缩规则
- 若后续调整某一类展示口径，必须同时同步对应模块文档，避免出现“同类数字在不同页面使用不同规则”的漂移。

## 字体与图标资源规范

- 全客户端 UI 字体真值为内置字体资源，不依赖 Windows/Android 系统字体回退。
- 当前正式字体资源为 `res://assets/fonts/SourceHanSansSC-VF.ttf`，随客户端一并分发。
- 顶部灵石、静音等符号型展示统一使用客户端内置 SVG 图标，不恢复到 emoji 方案。
- 若后续替换字体，必须同时确认授权、导出包含、真机显示与动态弹窗继承效果。

## 长屏与安全区契约

- 主界面逻辑设计基线固定为 `720×1280`，`1080×1920` 只是默认输出尺寸，不是新的摆控件基线。
- 主场景采用 `SafeTop / ContentFrame / SafeBottom` 三层结构：
  - `ContentFrame` 承载核心交互区，保持原 9:16 主构图。
  - `SafeTop`、`SafeBottom` 只负责吃掉刘海、灵动岛、Home Indicator 和超长屏新增空间。
- 长屏、异形屏和平板新增的高度/宽度，不得直接用来拉大主界面核心控件。
- 若新增主界面常驻控件，默认放入 `ContentFrame`；只有背景、装饰填充和可滚动延展区才能进入安全区语义。
- 当前人工长屏预览场景为 `res://scenes/debug/ResolutionPreview.tscn`，用于 `1080×1920`、`1080×2400`、`1125×2436` 三组分辨率的快速验收。
- 正常业务入口不得主动改运行窗口尺寸；窗口改尺寸逻辑只允许保留在调试场景中。

## 文档与代码一致性

- 文档描述“当前行为”，不保留历史行为并列表述。
- 每次改动提示策略、状态流、测试入口时，必须同步文档。
