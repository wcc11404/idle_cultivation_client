## 术语约定

- `reason_code`：服务端返回的稳定业务语义码。
- `reason_data`：客户端拼装文案和展示所需的结构化数据。
- 乐观更新：客户端先行更新本地展示，后续再上报或全量同步。
- 预扣池：炼丹每次开炉后，本地先预扣当前轮次材料和灵气。
- 回放：历练按服务端返回的战斗时间轴进行本地 UI 播放。

- 测试账号：`test / test123`
- 分辨率 / 长屏人工预览场景：`res://scenes/debug/ResolutionPreview.tscn`

## 默认开发规范

- 业务真值以“当前代码 + 当前测试 + 当前文档”一致后的结果为准。
- `docs/todo/` 只记录想法、规划和待办，不当作现状真值文档。
- `docs/todo/` 默认视为人类维护区域；除非人类明确要求，不要主动新增、修改或整理其中的文件。
- 命名规范统一如下：
  - 文件夹：小写加下划线
  - 代码/场景脚本文件：首字母大写
  - 函数、变量：小写加下划线
  - 常量：大写加下划线
- 业务提示统一走 `reason_code + reason_data`，不要恢复成直接透传服务端文案。
- UI 优先复用现有模板，不在模块里随手复制一套新样式。
- 数值显示优先复用 `res://scripts/utils/UIUtils.gd`，不要在模块里手写新格式。
- 主界面按 `720×1280` 设计基线维护；长屏只调安全区、背景和可伸缩容器，不随意拉散主构图。
- 改代码如果影响行为、测试入口、UI 模板、文案或显示规则，必须同步改文档。

## 最常见的同步关系

- 改模块逻辑：
  同步 `docs/02-modules/` 对应页。
- 改通用按钮 / 弹窗 / Tab / 展示面板：
  同步 `docs/04-maintenance/UI_Templates_Index.md`。
- 改测试入口、测试账号、运行方式：
  同步 `docs/03-testing/GUT_API_Testing.md`。
- 改长屏预览方式或安全区规则：
  同步 `docs/03-testing/UI_Resolution_Manual_Checklist.md` 与相关模块文档。

## 不确定时怎么判断

- 现状与旧注释/旧 todo 冲突：
  以当前代码和当前测试行为为准。
- 文档之间冲突：
  以更具体的模块文档为准，并回头修正总览或维护文档。
- 不确定某段旧逻辑是否还能删：
  先看 [瘦身与契约维护规范](./04-maintenance/SlimmingAndContracts.md)，不要直接猜。
