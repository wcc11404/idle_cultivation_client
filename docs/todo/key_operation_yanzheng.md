# BS架构优化
需求背景：目前已经初步将游戏改造成弱联网的形式，拆分出了独立的服务端和客户端，但是只在进入游戏和手动登出是，才需要与服务端通信，其他操作都在客户端本地进行，缺乏必要的安全验证和数据同步机制
优化目标：优化关键操作的验证机制，确保玩家数据的安全性和一致性，方案的设计要考虑改造成本，尽量减少因乐观更新带来的数据回滚等复杂操作，同时也要考虑到游戏的性能和用户体验，部分常用操作必须要做乐观更新
特别注意：不需要考虑网络失败或者实现客户端的操作队列机制，因为大部分操作都是非乐观更新的，服务端不确认客户端操作就不会继续进行

思考一下通信校验模式
- 非乐观更新模式
    - 第一种，客户端检验前置依赖条件，然后先计算预更新的数值，然后调用服务端，服务端检验前置依赖条件，然后用客户端的数值更新数据库，并返回给客户端是否成功，客户端根据返回结果，再实际更新本地数据；
    - 第二种，客户端检验前置依赖条件，然后调用服务端，服务端检验前置依赖条件+实际更新的数值，更新数据库后把要改动的数值返回给客户端，客户端根据返回结果，再实际更新本地数据；
    - 以上取舍，第一种需要实现预更新模式，第二种需要实现服务端更新模式，一部分逻辑放在了服务端实现，代码功能耦合
- 乐观更新模式
    - 客户端直接更新本地数据，记录增量数据，调用服务端，服务端检验前置依赖条件，然后用客户端的增量数值更新数据库，并返回给客户端是否成功，如果失败则回滚本地数据，如果成功没有变化；
- 队列机制
    - 因为有乐观更新，所以必须有排队模式按照先入先出的原则，乐观更新的操作（修炼+战斗）需要加入到队列中等待，一旦有非乐观更新的操作，需要先按顺序把乐观更新的操作同步服务端，再继续执行
    - 这里需要实现操作回调机制

## 通用更改
- 客户端
    - 取消每五分钟自动保存的功能，只保留设置页面，登出和保存按钮；
    - 无网络/通信失败处理，构建通用函数，在富文本框提示操作失败，请检查网络（只对游戏内的操作生效），累计失败10次，返回登录界面，入参可选是否在富文本框提示，也可以选择退出到登录界面还是退出游戏，特例，登录界面还保留原状不调用这个函数；
- 服务端
    - 需要依赖的配置文件全部软连客户端的配置文件，目前应该只软连了一部分；
    - 把game.py按照具体游戏功能再拆分，每个功能对应一个.py文件（例如，修炼/离线收益/突破，储纳（物品使用等），历练，术法，炼丹），load_game和save_game仍旧保留在game.py里；
    - 所有可供客户端调用api接口，都要支持in,out日志且需要打印在一行，这样方便后续通过account_id查找所有操作记录，日志格式：[IN/OUT] METHOD /path - account_id: xxx - data: {xxxx} - 耗时: 0.123s，其中account_id是请求头里的token解码后的，data里in日志放请求参数，out日志放返回参数，没有参数的情况，data为{}，耗时只有out日志有；
    - 服务端错误码， 200表示成功，设定一个其他值表示通用失败，401表示token无效，然后通信失败这个应该不用服务端返回；
    - 接口模板定义，请求头包括token，入参必须包括操作id和操作触发时间戳,操作id由客户端通过uuid.uuid4()生成，服务端根据操作id来判断操作是否重复，重复操作返回400错误码；

## 离线收益
- 服务端，服务端复用现有的计算离线收益接口，优化逻辑并更新相应数据库字段，返回离线收益数据和更新后的玩家数据，注意这里的计算逻辑要参考客户端的计算逻辑，灵气增加不能超过上限，灵气上限是个动态值，是由境界和其他一些信息汇总计算得到的
- 客户端，成功登录以后，客户端异步调用离线收益计算接口，如果服务端返回有离线收益数据，客户端直接根据返回的更新后的玩家数据，更新本地数值和ui并显示具体的离线收益，如果没有离线收益（收益为0的情况），不做任何处理，如果是通信失败等情况遵循上述无网络/通信失败处理

## 修炼
- 服务端，支持修炼校验接口，服务端计算修炼时长*灵气增加速度的值作为预估值，校验修炼效果是否处在合理区域（<=1.1倍的预估值）且，使用api提供的数据更新相应数据库字段，返回是否合法，入参包括灵气和吐纳术法的熟练度和气血的增量变化，修炼时长（秒）
- 客户端
开始修炼后，每秒结算一次，记录增量信息（包括灵气变化和吐纳心法的熟练度变化和气血的变化）和耗时（秒），如果服务端返回的合法，更新本地数值和ui，如果不合法不更新数值，如果是通信失败等情况遵循上述无网络/通信失败处理，不合法和通信失败则停止修炼；

## 突破
- 服务端，复用现有的突破api，校验玩家是否满足突破条件并更新相应的数据库字段，如果满足突破条件返回消耗的灵石、物品、灵气和突破后的等级，不满足返回缺少的灵石、物品、灵气
- 客户端，先校验灵石物品灵气是否足够，如果不足则复用现有的提示逻辑，如果足够，再调用突破api，如果返回突破成功，更新本地数值和ui，失败则和不足的提示一样，如果是通信失败等情况遵循上述无网络/通信失败处理；

## 储纳
- 服务端
    - 支持物品的use，open, drop接口，服务端验证合法性（物品数量是否足够，使用、打开、丢弃操作是否合法，物品有effect字段的代表可以使用，有content字段的代表可以打开），计算物品效果并更新相应数据库字段，返回需要客户端更新的数据；
    - 支持储纳整理接口，服务端验证合法性，计算最终效果并更新相应数据库字段，返回需要客户端更新的数据；
    - 支持储纳最大上限扩容接口，服务端验证合法性（目前什么都不消耗，最大扩容上限200，这个数值直接配置在配置文件里），计算最终效果并更新相应数据库字段，返回需要客户端更新的数据；
- 客户端
    - 用户点击物品详情面板相应的按钮后，主动调用相应接口（物品id+物品数量），根据返回数据更新本地数值和ui，不做乐观更新，不做批量操作功能（调用接口固定物品数量是1，预留扩展空间），如果返回操作失败，则没有任何处理，如果是通信失败等情况遵循上述无网络/通信失败处理；
    - 用户点击整理按钮后，主动调用整理接口，再根据返回数据更新本地数值和ui，不做乐观更新，如果返回操作失败，则没有任何处理，如果是通信失败等情况遵循上述无网络/通信失败处理；
    - 用户点击扩容按钮后，主动调用扩容接口，再根据返回数据更新本地数值和ui，不做乐观更新，如果返回操作失败，则没有任何处理，如果是通信失败等情况遵循上述无网络/通信失败处理；

## 历练
- 服务端
    - 支持普通区域的历练成功校验接口，应用随机数结算对应区域的掉落奖励，将技能熟练度增加的值，气血的变化等更新数据库相应字段，返回是否成功和奖励信息；
    - 支持无尽塔的成功通关接口，层数加一并判断发放奖励，将技能熟练度增加的值，气血的变化等更新相应数据库字段，返回是否成功和奖励信息；
    - 支持每日副本的查询接口和通关接口，复用现在的就可以，通关接口做一点改变，将技能熟练度增加的值，气血的变化等更新数据库相应字段，返回是否成功和奖励信息；
    - 支持历练失败的接口，将技能熟练度增加的值，气血的变化等更新数据库相应字段，返回是否成功；
- 客户端
    - 战斗中气血和术法熟练度等信息和ui在本地实时更新，存储增量值；
    - 战斗之间之间的等待时间，写进配置文件吧；
    - 每次战斗结束（包括失败和成功）将增量值、耗时、倍速、历练区域id, 层数等，作为参数调用相应的api，一定要阻塞等待api返回结果，再根据服务端返回的奖励信息，更新掉落物品和ui，如果返回失败，回滚气血和熟练度的变化，并结束历练，如果是通信失败等情况遵循上述无网络/通信失败处理，回滚气血和熟练度的变化并停止历练；
这里要注意，如果在历练过程中使用了补血丹等，补血丹的回复气血效果会被储纳模块实时更新到服务器，服务端对历练模块影响的气血变化的记录不是实时的，所以要记录增量，服务器计算最终值并存储到数据库中，历练对气血的增量影响有可能是增加也有可能是减少，客户端回滚也是回滚本地记录的增量值！！！

## 术法
- 服务端
    - 支持术法的装备、卸下接口，校验玩家数据合理性更新数据库相应字段，返回是否成功；
    - 支持术法的充能、升级接口，校验玩家数据合理性更新数据库相应字段，返回是否成功；
- 客户端
    - 逻辑改成战斗中不能装备、卸下、升级术法，但是可以充灵气；
    - 点击装备、卸下按钮，保留之前的逻辑（判断是否是战斗中和装备数量限制），主动调用接口，根据服务端返回的是否成功，更新本地数值和ui，不做乐观更新，如果返回失败，不做任何处理，如果是通信失败等情况遵循上述无网络/通信失败处理；
    - 点击充能，升级按钮，校验前置依赖资源是否足够（升级需要判断是否是战斗中），主动调用接口，再根据服务端返回的是否成功，更新本地数值和ui，不做乐观更新，如果返回失败，不做任何处理，如果是通信失败等情况遵循上述无网络/通信失败处理；

## 炼丹
- 服务端，支持炼丹校验接口，入参为丹方id，炼丹耗时，验证玩家资源是否足够，耗时是否在合理区间，计算成功率信息并判断结果，更新数据库相应字段，返回是否成功和物品变更信息；
- 客户端
更改炼丹逻辑，之前是预扣缴材料，然后失败返还材料成功则给物品，现在改成完成的一瞬间才扣减材料发放物品，如果失败则扣缴一半的材料，方便服务器计算；
点击开始炼丹按钮，复用之前的逻辑，每次炼丹完成时，主动调用接口，阻塞等待api返回结果，再根据服务端返回的是否成功，更新本地数值和ui，不做乐观更新，如果是通信失败等情况遵循上述无网络/通信失败处理，并停止炼丹；

## 定期同步机制
- 客户端，每10分钟，请求服务端load_game接口，根据服务端返回的玩家数据，更新本地数值和ui，但一定要在不修炼、不历练的状态调用，否则会表现出回滚的情况

### 一些废弃的逻辑
开始修炼后，每秒结算一次，更新本地数值和ui, 记录增量信息（包括灵气变化和吐纳心法的熟练度变化和气血的变化），每累计5次调用一次修炼校验接口，如果服务端返回的不合法，回滚增量更新，如果是通信失败等情况遵循上述无网络/通信失败处理，然后再继续累计5次调用一次修炼校验接口；
这里要注意，必须要有因为修炼导致的灵气或者熟练度或者是气血的增长才算一次累计，如果灵气或者气血或者吐纳术法的熟练度从非满值达到满值且有累计的次数，需要先调用一次修炼校验接口，如果涉及到消耗灵气的事件发生时（突破，充灵气，炼丹等）且有累计的次数，也要调用一次api, 如果停止修炼且有累计次数也要调用api
调用以后除了通信失败，其他合法和不合法的情况，累计次数都清零
