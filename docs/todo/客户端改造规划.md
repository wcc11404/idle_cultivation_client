# 客户端改造规划（服务端权威化）

## 1. 目标与原则

将客户端从“本地规则主导”改为“服务端权威主导”：

- 客户端：负责渲染、交互、临时缓存、必要乐观更新。
- 服务端：负责状态判定、资源扣发、战斗/炼丹/修炼结算、防作弊。
- 本地数据：只做 UI 快照与会话态，不作为最终真值。

核心原则：
1. 关键行为必须走 API。
2. 本地可乐观展示，但最终以后端返回覆盖。
3. 失败场景必须可回滚、可恢复、可提示。

---

## 2. 本轮范围

### 2.1 保留
- 注册/登录前置逻辑。
- 现有 UI 结构、富文本日志风格。
- 现有模块化页面组织。

### 2.2 改造
- 修炼、突破、背包、术法、炼丹、历练全部改为 API 权威。
- 本地 `scripts/core/*` 从“业务计算”转为“状态容器 + 渲染适配”。

### 2.3 明确不做
- 离线可玩后回放同步（本轮不支持）。
- 大规模 UI 重构。

---

## 3. 客户端分层与状态模型

- **UI 层**：`scripts/ui/*`
  - 仅负责事件、动效、文本、局部 loading。
- **编排层**（建议 `GameOrchestrator`，可先挂 `GameManager`）
  - 调 API、处理错误、更新本地快照、派发 UI 刷新。
- **网络层**：`NetworkManager` + `GameServerAPI`
  - 统一注入 `Authorization`、`operation_id`、`timestamp`。
  - 统一超时、重试、token 过期处理。
- **本地快照层**：`LocalPlayerSnapshot`
  - `server_state`：最近一次服务端确认状态。
  - `optimistic_state`：乐观临时态。
  - `pending_ops`：进行中的关键操作（修炼上报、炼丹预扣、战斗结算等）。

数据写入规则：
1. 先写 `optimistic_state`（如每秒修炼数字动画）。
2. API 返回后同步到 `server_state`。
3. UI 展示 = `server_state + 合法 optimistic 增量`。
4. 失败时按 `pending_ops` 回滚。

## 4. 配置软连接（客户端统一使用服务端 JSON）

目标：客户端 `scripts/core` 的配置 JSON 全部软连接到服务端 `app/modules`。

映射：
- `scripts/core/realm/realms.json` -> `app/modules/cultivation/realms.json`
- `scripts/core/inventory/items.json` -> `app/modules/inventory/items.json`
- `scripts/core/spell/spells.json` -> `app/modules/spell/spells.json`
- `scripts/core/alchemy/recipes.json` -> `app/modules/alchemy/recipes.json`
- `scripts/core/lianli/areas.json` -> `app/modules/lianli/areas.json`
- `scripts/core/lianli/enemies.json` -> `app/modules/lianli/enemies.json`
- `scripts/core/lianli/tower.json` -> `app/modules/lianli/tower.json`

注意：
- 软连接前备份旧文件。
- 导出环境若不支持符号链接，打包前做复制同步。

## 5. 分模块改造方案（已按最新确认项更新）

## 5.1 登录与初始化

流程：
1. `auth/login` 成功后保存 `token/expires_in/account_info/data`。
2. 用登录返回 `data` 初始化本地快照。
3. 调 `game/claim_offline_reward`。
4. 若有奖励，更新本地并输出富文本。
5. 进入主界面。

失败策略：
- 离线奖励失败不阻断进场。
- 鉴权失败按统一鉴权流程回登录。

## 5.2 内视（修炼/突破）

接口：
- `POST /api/game/player/cultivation/start`
- `POST /api/game/player/cultivation/report`
- `POST /api/game/player/cultivation/stop`
- `POST /api/game/player/breakthrough`

策略：
- start 成功后进入修炼态。
- 每秒本地乐观增长（仅展示）。
- 常规情况下每累计 5 次修炼上报一次 `report(count=5)`。
- 点击“停止修炼”时：
  - 若距离上次上报累计次数 `pending_count` 满足 `0 < pending_count < 5`，先执行 `report(count=pending_count)`，成功后再调 `stop`；
  - 若 `pending_count = 0`，直接调 `stop`。
- 点击“突破/使用补气丹/补血丹/其他会影响修炼结算的数据操作”前，先执行一次修炼结算：
  - 若 `pending_count > 0`，先 `report(count=pending_count)`；
  - 再执行目标操作，避免状态冲突。
- breakthrough 完全以后端返回覆盖本地（境界、灵气、血量、材料、背包）。

修炼结算串行化（建议实现轻量队列）：
- 增加一个 `cultivation_flush_and_then(action)` 统一入口。
- 所有“需要先结算修炼”的操作都通过该入口串行执行，保证顺序：
  1) flush pending 修炼增量；
  2) 执行目标 action；
  3) 用服务端返回统一覆盖本地。
- 不需要做复杂通用任务队列，只需“修炼结算闸门 + 单飞锁”（同一时刻仅一个 flush 请求）即可避免冲突。

失败处理：
- `report` 失败时不回滚本地修炼增量：将本次 `count` 继续累计到 `pending_count`，下次合并上报（例如 5 + 5 = `report(count=10)`）。
- `report` 时间异常同样走“累积后续再上报”策略，并提示同步异常。
- 连续失败阈值（如 3 次）后自动 stop 并提示“同步异常，已停止修炼”。

## 5.3 储钠（背包）

接口（确认）：
- `POST /api/game/inventory/use`（使用/打开都走这里）
- `POST /api/game/inventory/discard`
- `POST /api/game/inventory/organize`
- `POST /api/game/inventory/expand`
- `GET /api/game/inventory/list`

策略：
- 进入储钠页：先用本地快照预渲染，再异步 `list` 覆盖。
- 所有按钮均先发请求，成功后更新本地。
- 本地只维护 UI 选择态/排序态，不做最终扣发判定。

## 5.4 术法

接口（确认必须接入）：
- `POST /api/game/spell/equip`
- `POST /api/game/spell/unequip`
- `POST /api/game/spell/upgrade`
- `POST /api/game/spell/charge`
- `GET /api/game/spell/list`

策略：
- 进入术法页：本地预渲染 -> 异步 `spell/list` 覆盖。
- 升级/充灵气/装备/卸下均以后端返回覆盖。

## 5.5 炼丹房（含预扣设计）

接口：
- `POST /api/game/alchemy/start`
- `POST /api/game/alchemy/report`（失败自动重试 1 次）
- `POST /api/game/alchemy/stop`

预扣规则（按你要求）：
1. 点击开始后，本地按配方把材料放入 `alchemy_reserved_materials`（预扣池），从储钠可用数量中扣除。
2. 服务端实际扣除发生在 `alchemy/report` 成功时。
3. 每次 `report` 成功：
   - 根据 `materials_consumed` 正式扣账；
   - 从预扣池消解对应数量；
   - 产物入包，日志输出。
4. 停止炼丹（主动或异常）：
   - 调 `alchemy/stop`；
   - 将预扣池未消解材料全部返还可用背包。

关键点：
- 预扣池是“本地锁定”，不是“本地删除”。
- 任意异常退出都必须触发预扣回滚。

## 5.6 历练（重点）

接口（确认）：
- `POST /api/game/lianli/simulate`（必须传 `area_id`）
- `POST /api/game/lianli/finish`
- `GET /api/game/lianli/foundation_herb_cave`
- `GET /api/game/lianli/tower`

流程：
1. 进入历练选择页：本地预渲染次数/塔层，再异步刷新两项接口。
2. 点击区域：先最小本地校验（血量、次数、区域解锁）再 `simulate(area_id)`。
3. 用 `battle_timeline` 渲染战斗页。
4. 战斗结束或退出时 `finish(speed,index?)` 结算（失败自动重试 1 次）。
5. 连战：结算成功才允许下一场。

结算失败强制策略（重点）：
- 任意 `finish` 失败：
  1) 立即关闭自动战斗；
  2) 停止战斗播放；
  3) 清空本地当前战斗态；
  4) 返回历练选择 tab；
  5) 弹出错误提示（可重试/返回）。

幂等策略：
- 客户端不做重复调用去重，正常发送；
- 服务端用 `operation_id` 去重（你已确认）。

## 5.7 设置页

- 取消“保存按钮”（不再调用 `game/save`）。
- 保留登出按钮并调用 `POST /api/auth/logout`。
- `game/save` 服务端保留，但客户端本轮不调用。

## 5.8 三大流程互斥规则（修炼 / 炼丹 / 历练）

互斥约束：同一时间只能进行一个流程。

- 若“修炼中”点击开始炼丹或进入历练：提示“请先停止修炼”。
- 若“炼丹中”点击开始修炼或进入历练：提示“请先停止炼丹”。
- 若“历练战斗中”点击开始修炼或开始炼丹：提示“请先结束历练”。

实现建议：
- 统一在编排层维护 `active_mode in {none, cultivation, alchemy, lianli}`。
- 所有入口按钮先过 `can_enter_mode(target_mode)` 校验。
- UI 按钮态与 `active_mode` 联动禁用，避免用户连点触发冲突请求。

## 6. 统一错误分支与 UI 状态机（重点）

建议所有页面统一 4 态：
- `idle`：可操作
- `loading`：操作进行中（按钮禁用/局部转圈）
- `success`：短提示 + 刷新
- `error`：提示 + 回滚 + 恢复可操作

通用分支：
1. **网络错误（超时/断网）**：
   - 单次/偶发：保留当前页、操作按钮恢复、Toast「网络异常，请重试」、不清除 `server_state`。
   - 连续多次（建议阈值：同一会话连续 3 次关键请求网络失败）：
     - 立即停止当前流程（修炼/炼丹/自动战斗）；
     - 清理本地会话态（token 与 pending_ops）；
     - 强制登出并跳转登录页；
     - 提示「网络异常次数过多，请重新登录」。
2. **鉴权错误（401/Invalid token/KICKED_OUT）**：
   - 停止当前流程（修炼/炼丹/战斗）；
   - 清 token；
   - 跳转登录页。
3. **业务失败（success=false）**：
   - 保留页面；
   - 回滚本次 optimistic；
   - 展示后端 message/reason。
4. **关键流程失败（炼丹上报/历练结算）**：
   - 先自动重试 1 次；
   - 若重试仍失败，弹窗提示用户并执行“强一致收敛动作”（停止自动流程 + 清 battle/alchemy pending）。

## 7. Loading 交互建议（你问的网络不好场景）

建议“局部优先、全局兜底”：

- **局部 loading**（默认）：按钮转圈+禁用，适用于绝大多数接口。
- **全屏 loading**（仅关键切换）：登录、首次进主界面、战斗结算提交中。
- **超时阈值**：
  - 300ms 内不显示 loading（避免闪烁）；
  - >300ms 显示；
  - >8s 显示“网络较慢，可继续等待或重试”。

这样既不打断体验，也能在弱网时给明确反馈。

## 8. 错误码方案建议（你第7点）

建议服务端新增/统一字段：
- `error_code`（机器可判定）
- `message`（先保留，便于人读与日志定位；暂不作为客户端分支依据）
- `recoverable`（是否可重试；客户端本轮先不自动重试）

建议错误码分段：
- `AUTH_****`：鉴权类（如 `AUTH_TOKEN_INVALID`, `AUTH_KICKED_OUT`）
- `NET_****`：网络/网关类（客户端本地生成）
- `GAME_COMMON_****`：通用业务
- `CULT_****`：修炼
- `INV_****`：背包
- `SPELL_****`：术法
- `ALCH_****`：炼丹
- `LIANLI_****`：历练

最小可用清单（建议首批）：
- `AUTH_TOKEN_INVALID`
- `AUTH_KICKED_OUT`
- `CULT_REPORT_TIME_INVALID`
- `INV_ITEM_NOT_ENOUGH`
- `INV_CAPACITY_MAX`
- `SPELL_SPIRIT_NOT_ENOUGH`
- `ALCH_NOT_ALCHEMIZING`
- `ALCH_REPORT_TIME_INVALID`
- `LIANLI_ALREADY_BATTLING`
- `LIANLI_SETTLE_TIME_INVALID`
- `LIANLI_NOT_IN_BATTLE`

客户端分支只看 `error_code`，避免依赖中文 `message`。

## 9. 里程碑计划

### M1（基础）
- JSON 软连接。
- API 中间层统一（鉴权/operation_id/timestamp/错误处理）。
- 登录 + 离线奖励链路。

### M2（核心）
- 内视权威化。
- 储钠权威化（含 `expand/list`）。
- 术法权威化（含 `equip/unequip/list`）。

### M3（复杂）
- 炼丹循环 + 预扣池 + 回滚。
- 历练 `simulate/finish(area_id)` 全流程 + 失败强制收敛。

### M4（收口）
- 设置页改造（取消保存、保留登出）。
- 全局 loading 与错误分支统一。
- 联调与回归。

---

## 10. 验收标准（DoD）

- 关键资源最终与服务端一致。
- 客户端不再本地决定关键结算结果。
- 弱网/超时/鉴权失败可恢复，且不留下脏状态。
- 历练结算失败会强制退出自动战斗并回选择页。
- 炼丹停止/异常会返还未消解预扣材料。

---

## 11. 执行原则

- 先打通链路，再优化体验。
- 优先保证权威正确性，再做流畅性。
- 能复用 UI 的地方尽量复用，避免过度重构。
