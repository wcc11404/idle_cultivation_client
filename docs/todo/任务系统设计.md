# 任务系统设计（每日任务 + 新手任务）

## 1. 目标与范围

- 在「地区」页的云稷城新增任务系统入口，进入任务界面后支持两个子页签：
  - 每日任务
  - 新手任务
- 任务进度、完成判定、领奖判定全部由服务端权威控制。
- 客户端负责展示、点击请求、文案反馈，不做本地业务真值。

本期先实现你指定的任务内容与奖励，不做额外复杂任务链。

---

## 2. 入口与命名

### 2.1 地区页入口

- 位置：地区页 -> 云稷城分组下新增任务按钮。
- 入口命名建议：`仙务司`（可后续再改文案，不影响接口）。

### 2.2 进入后页面

- 页面名称建议：`仙务司`。
- 顶部：返回按钮 + 子 TabBar（复用已封装模板）。
- 子页签：
  - `每日任务`
  - `新手任务`

---

## 3. UI 与交互设计

## 3.1 页面结构

- 顶部区域：
  - 左侧返回按钮（复用现有返回按钮样式模板）
  - 中间标题（仙务司）
  - 下方子 TabBar（每日任务 / 新手任务）
- 列表区域：
  - 两个 Tab 都支持纵向滚动
  - 不显示滚动条（可滚动但滚动轴隐藏）

## 3.2 单个任务卡片（复用 DisplayPanel）

- 基础结构（从上到下）：
  1. 标题：任务名
  2. 描述：任务说明（一行 label）
  3. 进度条：显示当前进度
- 右侧独立列：
  - `领取`按钮（绿色模板）
  - 未完成时：灰色不可点击
  - 已完成未领取时：绿色可点击
  - 已领取时：灰色不可点击（文字可为“已领取”）

## 3.3 排序规则

- 主排序：按任务配置顺序（`sort_order`）。
- 状态分组：已领取任务整体在未领取任务下方。
  - 即：未领取（含未完成/可领取）在上，已领取在下。
  - 各分组内仍按 `sort_order` 排序。

## 4. 服务端设计

## 4.1 新增模块

建议新增 `task` 模块（与现有模块同级）：

- `TaskData`：读取任务配置（每日 + 新手）
- `TaskSystem`：玩家任务状态与进度管理
- `TaskApi`：对外接口

## 4.2 玩家侧数据结构（建议）

- `daily_tasks`：`{task_id -> task_state}`
- `newbie_tasks`：`{task_id -> task_state}`
- `task_state` 建议字段：
  - `progress: int`
  - `completed: bool`
  - `claimed: bool`

## 4.3 接口设计（建议）

### 1) 查询任务列表

- `GET /api/game/task/list`
- 返回：
  - `daily_tasks`: 列表
  - `newbie_tasks`: 列表
  - 每个任务项包含：
    - `task_id`
    - `name`
    - `description`
    - `task_type` (`daily` / `newbie`)
    - `progress`
    - `target`
    - `completed`
    - `claimed`
    - `sort_order`
    - `rewards`

### 2) 领取任务奖励

- `POST /api/game/task/claim`
- 入参：
  - `task_id`
- 结果：
  - 成功：发奖并标记 `claimed=true`
  - 失败：未完成 / 已领取 / task_id 无效

reason_code 建议（语义码风格）：
- `TASK_LIST_SUCCEEDED`
- `TASK_CLAIM_SUCCEEDED`
- `TASK_CLAIM_TASK_NOT_FOUND`
- `TASK_CLAIM_NOT_COMPLETED`
- `TASK_CLAIM_ALREADY_CLAIMED`

## 4.4 进度结算挂点（服务端）

以下行为成功结算后推进任务进度：

- 修炼10次（每日）
  - 挂点：`cultivation/report` 成功结算整秒时按“有效修炼秒数”累计，换算为“次”。
- 战斗10次（每日）
  - 挂点：历练战斗结算成功（`lianli/finish` 成功结算一场）。
- 炼丹1次（每日）
  - 挂点：炼丹单次产出结算成功（`alchemy/report` 成功轮次）。
- 采集5次（每日）
  - 挂点：采集上报成功（`herb/report` 成功轮次）。
- 术法充灵5次（每日）
  - 挂点：`spell/charge` 成功。

通用进度规则：
- 任务进度达到目标后不再继续增长（封顶到 target）。
- 界面进度始终按整数显示（例如 `7 / 10`、`10 / 10`）。

新手任务（一次性）：
- 打开新手礼包Ⅰ / Ⅱ / Ⅲ
  - 挂点：`inventory/use` 成功且使用对应礼包 item_id 时推进对应任务。

## 4.5 每日重置与登录重置规范（统一约定）

为避免逻辑分散，建立统一规范：

### A. 登录重置（login/logout reset）

适用：有“运行中状态”的系统，例如：
- 修炼（is_cultivating）
- 历练战斗（is_battling）
- 炼丹（is_alchemizing）
- 采集（is_gathering）

要求：
- 每个系统提供自己的登录重置函数（如 `reset_runtime_state_on_login_logout`）。
- 登录、登出都调用同一函数，避免重复实现和遗漏。

### B. 每日重置（daily reset）

适用：
- 每日副本次数
- 每日任务状态

要求：
- 每个每日系统提供独立每日重置函数（如 `reset_daily_state`）。
- 登录时统一做“是否跨天”检查，跨天后调用每日重置函数。
- 新手任务不参与每日重置，进度永久保留。

## 6. 客户端逻辑设计

## 6.1 模块建议

新增 `TaskModule.gd`（或 `XianwuModule.gd`）：

- `show_tab(task_type)`：切换每日/新手
- `request_task_list()`：拉取服务端任务数据
- `render_task_cards(tasks)`：渲染任务卡片
- `on_claim_pressed(task_id)`：领奖请求

## 6.2 客户端状态

- 本地只缓存当前列表展示数据（非权威）。
- 每次领奖成功后，建议：
  - 重新拉一次任务列表，避免本地状态漂移。
  - 同步刷新玩家资产显示（灵石/仙晶等）。

## 6.3 文案策略

- 按 reason_code 映射固定文案，不透传服务端 message。
- 未完成按钮灰态，不额外弹提示（点击无效）。

## 6.4 顶部货币区（TopBar）文档约束

- 目标节点：`/root/Main/ContentFrame/VBoxContainer/TopBar/TopBarContent/SpiritStoneContainer`
- 命名调整（文档约定）：
  - `SpiritStoneContainer` 改为英文通用命名：`CurrencyContainer`
- 布局约束：
  - 货币区内容与 TopBar 顶边不保留额外边距（贴齐顶部布局规则）。
- 展示要求：
  - 在现有灵石实时显示基础上，新增仙晶数量实时显示（与玩家资产同步刷新）。
